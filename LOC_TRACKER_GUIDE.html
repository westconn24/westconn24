Benmore LOC Tracker

Lines-of-Code Tracking System — Team Guide
Automated · Private · Zero Daily Effort
Overview

What It Does

Automatically tracks each developer's lines of code output per day by scanning merged GitHub pull requests. Numbers update every morning with no manual effort after the initial setup.

What It Tracks

  • Lines added and deleted per day, per PR
  • Code files only — .md, .rst, .txt, .csv excluded
  • Refactor-heavy PRs filtered automatically (>15k deletions ignored by default)

Privacy Model

All data lives in each developer's private {username}/loc-data repo. Only accessible to people with a GitHub PAT that has repo scope.

Use as a Signal

Good for spotting productivity gaps. Not intended for ranking people — it does not capture review time, design, planning, or async communication work.

How It Works

Automated Pipeline

1

GitHub Action fires at 6 AM UTC

Runs inside each developer's private {username}/loc-data repo on a daily schedule.

2

Scans all merged PRs from yesterday

Pulls data across every repo the developer has access to.

3

Strips doc and data files

Fetches file-level stats to exclude non-code file types from the count.

4

Saves results to loc-log.json

Stored in the private loc-data repo — only reachable via authorized PAT.

5

CLI reads the file on demand

Run loc.py locally — it fetches the JSON using your GH_LOC_TOKEN PAT.

Fresh Setup — New Developer (~5 min)
1

Create a Personal Access Token (PAT)

Go to github.com/settings/tokens → Generate new token (classic)

  • Note: loc-tracker
  • Expiration: No expiration
  • Scope: check the repo box (top-level)
  • Click Generate token → copy it (starts with ghp_)
2

Create your private loc-data repo

gh repo create YOUR_USERNAME/loc-data --private
3

Copy the tracker files into it

gh repo clone YOUR_USERNAME/loc-data ~/YOUR_USERNAME-loc-data && cd ~/YOUR_USERNAME-loc-data
mkdir -p scripts .github/workflows
curl -o scripts/track_loc.py https://raw.githubusercontent.com/westconn24/westconn24/main/scripts/track_loc.py
curl -o .github/workflows/track-loc.yml https://raw.githubusercontent.com/westconn24/westconn24/main/.github/workflows/track-loc.yml
git add scripts/track_loc.py .github/workflows/track-loc.yml
git commit -m "Add LOC tracker"
git push
4

Add your PAT as a repo secret

Go to: github.com/YOUR_USERNAME/loc-data/settings/secrets/actions

Click New repository secret

  • Name: GH_PAT
  • Value: paste the ghp_... token

Click Add secret

5

Run the workflow — backfill last 30 days

gh workflow run track-loc.yml --repo YOUR_USERNAME/loc-data --field days_back=30

Takes ~1–2 minutes to complete.

6

Add token and CLI to your shell

# Add token to shell (replace with your actual token)
echo 'export GH_LOC_TOKEN=ghp_your_token_here' >> ~/.zshrc && source ~/.zshrc

# Install the CLI
curl -o ~/loc.py https://raw.githubusercontent.com/westconn24/westconn24/main/loc.py
CLI Reference

Flags

--user USERNAME Target developer — last 7 days by default --days 30 Extend the lookback window --week Summary only, no daily breakdown -v Show individual PRs per day --max-deletions N Refactor filter threshold (default 15000, 0 = off) --exclude REPO Exclude a specific repo from totals --token ghp_... Pass token explicitly instead of env var

Viewing the Team

python3 ~/loc.py --user westconn24
python3 ~/loc.py --user ArkashJ
python3 ~/loc.py --user aniture

You can view someone's stats as long as your GH_LOC_TOKEN has repo scope access to their loc-data repo. Shared Benmore org membership covers this automatically.

Maintenance

Zero Ongoing Effort

  • Runs automatically every morning at 6 AM UTC — no action needed
  • PAT never expires if set to "No expiration" at creation time
  • Manual backfill any range:
    gh workflow run track-loc.yml --repo USERNAME/loc-data --field days_back=N

If Numbers Drop to Zero

The developer's GH_PAT secret likely expired. To fix:

  • Regenerate the token at github.com/settings/tokens
  • Update the secret in their loc-data repo settings under Actions secrets
Limitations

What It Does Not Track

  • Direct pushes to main — no PR means no data captured
  • Code review time — reviewing PRs is invisible to this system
  • Design, planning, async communication — non-code contributions not captured

Use as a productivity signal — good for spotting gaps, not for ranking people.
